# Changelog

All notable changes to `standards` follow [Semantic Versioning](https://semver.org/) and are recorded here in [Keep a Changelog](https://keepachangelog.com/) format.

Each STEM repo declares the **Standard version** it follows in its top-level `CLAUDE.md`. `state/repos.md` mirrors those declarations so a single glance shows which repos lag.

## Versioning rules

- **Major** ‚Äî breaking change to a standard or template that forces adopters to migrate or pin an older version.
- **Minor** ‚Äî new standard or new template, or a non-breaking change to an existing one.
- **Patch** ‚Äî bug fixes that restore intended behaviour, plus typos, clarifications, and internal refactors. No change to documented contracts.

The version number is the git tag (`v1.0.0`, `v1.1.0`, ‚Ä¶). There is no version field inside any single file ‚Äî `git describe` is the source of truth.

Historical entries from `v1.0.0` through `v1.3.3` were written while this repo was bundled with the agent-config under the name `llm-settings`. From this point forward `standards` is scoped to the STEM standards alone; the agent-config wiring lives in [`luca-veronelli-stem/llm-settings`](https://github.com/luca-veronelli-stem/llm-settings) (unversioned, HEAD-only).

## [1.15.0] - 2026-06-03

### Changed
- `shared/standards/TESTING.md` gains an **Unattended-only test suites** section: the `tests/` project holds only tests that run to completion with no human intervention. A test that needs a human to act *during* the run (press a button, unplug a cable, observe a screen) must not live in the suite as a `Skip`-by-default case ‚Äî an unconditional `[<Fact(Skip = "Manual ‚Ä¶")>]` literal fakes coverage (inflates the skipped count while asserting nothing on any run), rots (nobody remembers to remove the `Skip` and perform the dance), and can't be gated cleanly (the literal needs a source edit to ever run, while a bare env gate with no `Skip` would let an unattended run hang waiting on the operator). Resolve it by (1) automating the human away with a fixture (programmable USB power switch / relay, input injector) so it becomes an unattended `[<HardwareFact>]`, or (2) demoting it to a manual step in a release / re-vendor runbook ‚Äî the state-machine logic such a scenario exercises belongs in fast fake-driven tests regardless; only the irreducibly-physical leg is a runbook candidate. The one exception is an **attended, env-gated** test (`[<ManualHardwareFact>]` keyed on an interactive env var, e.g. `BPT_HARDWARE_INTERACTIVE=1`) for the rare assertion of genuinely-physical, non-deterministic behavior that can be neither automated nor faithfully captured in prose ‚Äî dormant in every unattended run (CI never hangs), runnable on demand with no source edit, and explicitly distinct from a `Skip`-by-default literal. Complements the existing `CI.md` ‚Üí "Hardware-test exclusion" rule that already forbids substituting `Skip` for the `Category=Hardware` trait filter; the `[<HardwareFact>]` / `[<ManualHardwareFact>]` attributes are the reference implementation, originating in [`button-panel-tester#142`](https://github.com/luca-veronelli-stem/button-panel-tester/issues/142) (the `PhysicalUnplug‚Ä¶` deletion plus `PhysicalReplug‚Ä¶` re-gate that motivated this principle). Doc-only and additive ‚Äî nothing previously compliant becomes non-compliant on re-roll (standards docs are advisory, not analyzer-enforced ‚Äî see `MIGRATION.md` ‚Üí "Choosing the bump level"), so this is a minor; TESTING's stability marker stays at `v1.0.0`. Closes [#126](https://github.com/luca-veronelli-stem/standards/issues/126).
- `shared/standards/MIGRATION.md` gains a `Rollout phase for v1.15.0` section: single-commit per-repo bump PR (re-run `apply-repo-standard.ps1 -StandardVersion v1.15.0` ‚Üí bump `CLAUDE.md` `**Standard version:**` ‚Üí update `state/repos.md`), with the note that the only diff is the refreshed `docs/Standards/TESTING.md` inline copy plus the version stamps the rollout refreshes, and that adopters carrying a `[<Fact(Skip = "Manual ‚Ä¶")>]` human-in-the-loop test should resolve it per the new principle in a follow-up PR (same shape as the v1.10.0 note about removing `Skip = "...#NNN"` hardware workarounds).

## [1.14.2] - 2026-06-03

### Fixed
- `.github/workflows/dotnet-ci.yml` (reusable body): a transient `actions/cache` restore failure no longer skips Restore/Build/Test and reds `main` with no real test failure. A GitHub Actions step `if:` that contains no status check function (`success()`, `failure()`, `always()`, `cancelled()`) carries an **implicit `success()`**, so when the `Cache NuGet packages` step errored on a cache-service hiccup its `failure` conclusion flipped `success()` to false and every downstream step lacking a status function ‚Äî Restore, Build, Test ‚Äî was skipped; no `.trx` was produced and `dorny/test-reporter` then failed with "No test report files were found". Three coordinated changes. (1) `continue-on-error: true` on the `Cache NuGet packages` step ‚Äî a cache restore is an optimization, never a correctness gate, so the failed step's *conclusion* is forced to `success` (its `outcome` still records the real `failure`) and Build/Test run against a cold cache instead of being skipped. (2) The same guard on the `Cache Lean toolchain` step: reasoning about the step graph (issue item 2) showed it has the identical defect ‚Äî it sits before Restore/Build/Test on the Linux leg, so a transient elan-cache error would skip them just the same. (3) The `Test report` step now keys off the test step's own `outcome` (via new `id: test_xplat` / `id: test_full` handles) ‚Äî `if: ${{ always() && (steps.test_xplat.outcome != 'skipped' || steps.test_full.outcome != 'skipped') }}` ‚Äî instead of the mere presence of a `.trx`: it runs whenever the OS-relevant test step actually executed (success or failure, so genuine failures are still reported) and skips only when both test steps were skipped (an upstream failure such as Build aborted the run before any test ran), so a real upstream red is no longer masked by the reporter's own "No test report files were found" failure. First incidents on `button-panel-tester` (consuming `dotnet-ci.yml@v1.12.0`): run `26628708854` (2026-05-29, push of the #161 merge) and PR #178 (2026-06-03, a `.github/dependabot.yml`-only change) ‚Äî both Windows-leg failures at ~21s cleared by a no-op re-run, confirming transient cache, not a real failure. Patch: a bug fix restoring intended behaviour (the cache was always meant to be an optimization, never a gate) with no change to any documented contract ‚Äî same shape as `v1.5.3`'s reusable-`dotnet-ci.yml` fix. Adopters pick it up by bumping their `uses: ‚Ä¶/dotnet-ci.yml@vX` pin to `@v1.14.2` (per-repo MIGRATION bump; consumer side tracked by [`button-panel-tester#162`](https://github.com/luca-veronelli-stem/button-panel-tester/issues/162)). Closes [#123](https://github.com/luca-veronelli-stem/standards/issues/123).

## [1.14.0] - 2026-06-03

### Fixed
- `.github/workflows/mirror-bitbucket.yml` (reusable body) + `shared/templates/.github/workflows/mirror-bitbucket.yml` (caller stub): the mirror workflow silently dropped tags. It triggered only on `main` branch pushes and pushed a single explicit refspec (`git push bitbucket HEAD:refs/heads/main`), which carries no tags ‚Äî so release/version tags pushed to GitHub never reached the Bitbucket mirror. Two coordinated changes fix it. (1) The caller stub gains `on.push.tags: ['v*.*.*']`, so version-tag pushes also trigger the mirror. (2) The reusable body branches on `github.ref_type` (a reusable workflow inherits the caller's event context, so `ref_type` is `branch` or `tag`): a branch push to `main` runs `git push --follow-tags bitbucket HEAD:refs/heads/main`, mirroring the commit plus the annotated tags reachable from `main` that the mirror is missing; a tag push runs `git push bitbucket "$REF:$REF"`, mirroring only the pushed tag ref and never force-updating `bitbucket/main` from the detached tag checkout (pushing `HEAD:refs/heads/main` there would force the mirror's `main` back to the tagged commit). Backward-compatible workflow change ‚Üí minor, same shape as `v1.10.0`'s `category-filter`: the reusable-body fix propagates to adopters automatically on their next mirror run via the `@vX.Y.Z` pin, and the additive stub trigger is picked up on the next `apply-repo-standard.ps1` re-roll. End-to-end live verification on `button-panel-tester` is deferred to the first adopter bump (it requires this tag to exist and the stub re-rolled to it ‚Äî a per-repo MIGRATION bump). Closes [#122](https://github.com/luca-veronelli-stem/standards/issues/122).

### Changed
- `shared/standards/CI.md`: the "Workflows shipped per repo" table updates the Mirror row's trigger to `push to main, tag push (v*.*.*)`, and the "Mirror workflow" section documents the two `github.ref_type` push paths, the annotated-vs-lightweight decision (`--follow-tags` carries annotated tags only ‚Äî which the annotated release-tag convention satisfies, so `v*.*.*` tags flow through on the next `main` push; `--tags` carries all tags, for repos that deliberately mirror lightweight tags), and the first-run backfill behaviour. Doc-only ‚Äî no contract change to the CI standard; stability marker stays at v1.0.0.
- `shared/standards/MIGRATION.md` gains a `Rollout phase for v1.14.0` section: single-commit per-repo bump PR that re-rolls the mirror stub (the only diff is the new `on.push.tags` trigger plus the `uses:` pin bump to `@v1.14.0`), the note that the first `main` push after the bump backfills annotated tags reachable from `main` and that lightweight/unreachable tags need a one-time manual `git push ... --tags`, and the deferred `button-panel-tester` end-to-end verification.

## [1.13.0] - 2026-05-29

### Changed
- `shared/standards/MIGRATION.md` gains a **Choosing the bump level** section codifying the major/minor/patch decision in adopter-contract terms rather than by diff magnitude. The rule: after an adopter bumps the `**Standard version:**` pin and re-runs `apply-repo-standard.ps1` (the *bump-and-reroll*), is previously-compliant code/config *forced* to change ‚Äî or does the build break ‚Äî with no opt-in? Yes ‚Üí major; no ‚Üí minor/patch. The section enumerates the major triggers (tightening/reversing a rule so compliant code becomes a violation; adding a consumed `BannedSymbols.txt` entry; changing an enforced default such as the TFM `net10.0 ‚Üí net11.0` or an archetype's default visibility posture; renaming/removing a standard or template; changing the `docs/Standards/` layout or the `.stem-standard.lock` format; redefining an archetype's required project shape) and grounds the minor/patch side in real releases (new standard `v1.5.0`/`v1.9.0` = minor; additive guidance to an existing standard `v1.12.0`/[#94](https://github.com/luca-veronelli-stem/standards/issues/94) = minor; backward-compatible workflow change `v1.10.0` `category-filter` / `v1.8.0` self-extract = minor; the `v1.5.1`/`v1.4.0`/`v1.2.1` fixes = patch). The reasoning previously lived only scattered across CHANGELOG entries and was re-derived per release ‚Äî the #94 review mistook a high minor (`v1.11.0`/`v1.12.0`) for "overdue for a major" when SemVer here is event-driven and nothing breaking had ever shipped. `README.md`'s "Versioning" section cross-links the new section and states the bump-and-reroll rule inline. Doc-only and additive ‚Äî it documents the existing major/minor/patch contract rather than changing it, and nothing previously compliant becomes non-compliant, so this is itself a minor. ## [1.11.0] - 2026-05-29

### Changed
- `shared/templates/.github/dependabot.yml`: split the single `avalonia` group (`patterns: ["Avalonia*"]`) into `avalonia-runtime` (`patterns: ["Avalonia", "Avalonia.*"]`, `exclude-patterns: ["Avalonia.FuncUI*"]`) and `avalonia-funcui` (`patterns: ["Avalonia.FuncUI*"]`), and added `update-types: [minor, patch]` to every NuGet group (`avalonia-runtime`, `avalonia-funcui`, `testing`, `microsoft-extensions`). Two structural problems, one fix. (1) FuncUI has its own release cadence and its own compat dependency on Avalonia versions, so coupling it with the Avalonia runtime in one group is wrong ‚Äî a FuncUI bump should never ride with an Avalonia-runtime bump. (2) With no `update-types` filter, a grouped PR could bundle a low-risk minor/patch (e.g. `Avalonia.Diagnostics 11.3.7 ‚Üí 11.3.16`) with a high-risk major (`Avalonia 11 ‚Üí 12`), forcing the whole bundle to fail CI and sit red. Restricting each group to `minor + patch` keeps major-version bumps as standalone PRs (or `@dependabot ignore`-able directives), so a safe patch is no longer blocked behind a breaking major. Motivated by [`button-panel-tester` PR #123](https://github.com/luca-veronelli-stem/button-panel-tester/pull/123) (closed 2026-05-25, deferred to a dedicated Avalonia 12 migration ticket): that PR bundled FuncUI `1.5.1 ‚Üí 1.6.0` (minor) with Avalonia `11.3.7 ‚Üí 12.0.3` (major) plus `Avalonia.Diagnostics` patches; the whole PR turned red on FuncUI/Avalonia API incompatibility and the patch-level bumps that could have merged on their own were blocked. Adopted repos pick up the new grouping on their next `apply-repo-standard.ps1` run ‚Äî the rollout overwrites `.github/dependabot.yml` from the template, no source-code change required. The standards repo's own `.github/dependabot.yml` is **not** affected: it watches `github-actions` only (the repo has no `.csproj`/`.fsproj` for the NuGet ecosystem to walk) and its single `gha-minor-and-patch` group already carries `update-types: [minor, patch]`, so the dogfood already embodies the new policy. `shared/standards/MIGRATION.md` gains a `Rollout phase for v1.11.0` section. Closes [#114](https://github.com/luca-veronelli-stem/standards/issues/114).

## [1.9.0] - 2026-05-22

### Added
- New `APP_DATA` standard at `shared/standards/APP_DATA.md` codifying `<LocalApplicationData>\Stem\<AppName>\` as the per-user on-disk root for every STEM desktop app's logs, caches, DPAPI credentials, SQLite databases, and any future per-user configuration overrides. The convention pins the root to `Environment.GetFolderPath(Environment.SpecialFolder.LocalApplicationData)` (cross-platform ‚Äî resolves to `%LocalAppData%` on Windows, `~/.local/share` on Linux, `~/Library/Application Support` on macOS), the company segment to PascalCase `Stem` (matches the namespace prefix `Stem.Communication`, `Stem.ButtonPanelTester.GUI`, and Microsoft's canonical `CompanyName\AppName` guidance), and the app segment to a single PascalCase token with no `Stem.` prefix (`Stem\DeviceManager\`, `Stem\ButtonPanelTester\`, `Stem\DictionariesManager\`, `Stem\ProductionTracker\`, `Stem\SparkLogAnalyzer\` ‚Äî maps cleanly onto each repo's name). Sub-purpose folders `logs/`, `cache/`, `credentials/`, `db/` are mandatory as soon as a second data type lands at the per-app root. Replaces four divergent conventions surveyed across the stack on 2026-05-22: (1) `AppContext.BaseDirectory\logs\` in `stem-device-manager` ‚Äî root cause of the v0.4.1 silent-Excel-fallback bug that surfaced this work, broken under Program Files / single-file publish / thumb-drive deployment; (2) `%LocalAppData%\Stem.ButtonPanel.Tester\` in `stem-button-panel-tester-fsharp-core` (dotted, flat); (3) `%LocalAppData%\Stem.ButtonPanelTester\` in `button-panel-tester` (single-segment, flat); (4) `%AppData%\STEM\<AppName>\` in `stem-dictionaries-manager` and `stem-production-tracker` (Roaming, two-segment, ALL-CAPS ‚Äî binary blobs and DPAPI files have no business roaming via AD profile sync). All four collapse to one shape on next bump. Standard ships with a forbidden-alternatives table covering `AppContext.BaseDirectory`, `%AppData%` Roaming, `%ProgramData%`, and hardcoded absolute paths, each with the failure mode that motivates the ban. Closes [#109](https://github.com/luca-veronelli-stem/standards/issues/109). Companion follow-up [#110](https://github.com/luca-veronelli-stem/standards/issues/110) tracks extracting the path-resolution helper into a `Stem.AppData` NuGet once a second consumer earns its keep ‚Äî until then, inline copy-paste is fine. Bumps the standards count from nineteen to twenty.
- `shared/standards/APP_DATA.md` separates the forever surface from the transient migration kit explicitly. The forever piece is `StemAppData` (~15 LOC of `Path.Combine` sugar over `Environment.SpecialFolder.LocalApplicationData`, `GetAppRoot` / `GetLogsDir` / `GetCacheDir` / `GetCredentialsDir` / `GetDbDir`) ‚Äî every adopter keeps this long-term, optional only because `Path.Combine(localAppData, "Stem", appName, "logs")` at the call site is equally fine. The transient piece is `StemAppDataMigration` (~20 LOC) plus a `.appdata-version` schema-marker file at the per-app root ‚Äî both deletable one or two release cycles after the v1.9.0-aligned release, once the adopter's installed base has rolled over. The marker file left on disk after the helper is deleted is inert; no future code reads it. Greenfield adopters with no installed base skip the transient kit entirely. The split is load-bearing: it makes clear that v1.9.0 is not asking every STEM app to carry forever migration state on disk ‚Äî only the cutover from the four legacy roots carries any state, and only for as long as the adopter's installed base needs it. Both helpers shown in C# and F# forms; `button-panel-tester`'s [`CompositionRoot.fs:71-73`](https://github.com/luca-veronelli-stem/button-panel-tester/blob/main/src/ButtonPanelTester.GUI/Composition/CompositionRoot.fs#L71-L73) is cited as the F# adopter pattern, with the only delta against v1.9.0 being the folder shape (`Stem\ButtonPanelTester\` two-segment instead of `Stem.ButtonPanelTester\` flat).
